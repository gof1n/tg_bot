# Деплой на Railway

Telegram-бот (User + Admin) запускается как **worker** (без веб-сервера). База SQLite хранится на **Volume**, чтобы не терялась при редеплое.

**Важно:** при первом запуске база **пустая** — товары в каталоге и статистика по заказам появятся только после нажатия в админ-боте кнопки **«Обновить базу»** (или «Очистить и загрузить заново»). До этого каталог будет пуст, статистика — нулевая.

---

## 1. Создание проекта в Railway

1. Зайдите на [railway.app](https://railway.app), войдите через GitHub.
2. **New Project** → **Deploy from GitHub repo** → выберите репозиторий с этим проектом.
3. Railway сам определит Python и установит зависимости из `requirements.txt`.
4. В настройках сервиса задайте **Start Command**: `python main.py` (или оставьте пустым — тогда используется команда из Procfile: `python main.py`).

---

## 2. Переменные окружения

В Railway: **Service → Variables** добавьте все переменные из `.env.example`:

| Переменная | Описание |
|------------|----------|
| `USER_BOT_TOKEN` | Токен пользовательского бота (BotFather) |
| `ADMIN_BOT_TOKEN` | Токен админ-бота |
| `ADMIN_CHAT_ID` | ID чата/админа (одно число или несколько через запятую) |
| `DATABASE_PATH` | **Обязательно** путь к БД на Volume (см. ниже) |
| `CSV_URL` | Ссылка на Google Sheets (CSV) для синхронизации |

**Важно:** не загружайте файл `.env` в репозиторий — только задайте переменные в панели Railway.

---

## 3. Volume: чтобы база не удалялась

Без Volume при каждом редеплое контейнер пересоздаётся и файл `augsburg_liquid.db` пропадает. Volume — это постоянный диск, который монтируется в контейнер.

### Шаги в Railway

1. В вашем **Service** откройте вкладку **Volumes** (или через **+ New** → **Volume**).
2. Нажмите **Add Volume** и привяжите его к этому сервису.
3. Укажите **Mount Path**: например `/data` (папка, в которой будет храниться БД).
4. В **Variables** добавьте или измените:
   ```bash
   DATABASE_PATH=/data/augsburg_liquid.db
   ```
   Теперь бот будет создавать и читать базу по пути `/data/augsburg_liquid.db`, который лежит на Volume и сохраняется между деплоями.

### Как это работает

- **Build** (сборка): Volume не монтирован, ставить путь типа `/data/...` в build-командах нельзя.
- **Runtime** (запуск): при старте контейнера Volume монтируется в указанный Mount Path (`/data`). Скрипт `main.py` запускается и создаёт/открывает БД по `DATABASE_PATH=/data/augsburg_liquid.db`.
- При следующем деплое новый контейнер подключает тот же Volume — данные остаются.

### Проверка

После первого деплоя зайдите в админ-бота, нажмите «Обновить базу» или «Очистить и загрузить заново» — база создастся на Volume. После повторного деплоя заказы и пользователи должны остаться.

## 4. Мониторинг Volume

- В Railway: **Service → Volumes** — видно подключённый Volume и Mount Path.
- Размер и использование диска отображаются в панели Volume.
- Логи сервиса: **Deployments → выберите деплой → View Logs**. Убедитесь, что в логах нет ошибок типа «readonly» или «permission denied» для `DATABASE_PATH` — тогда запись идёт на Volume.

Если база не появляется или сбрасывается — проверьте, что:
1. Volume привязан к **этому** сервису.
2. `DATABASE_PATH` в Variables точно совпадает с путём внутри Mount Path (например `/data/augsburg_liquid.db` при Mount Path `/data`).
3. Перезапустили сервис после добавления/изменения Volume и переменных.

---

## 5. Итог

| Действие | Где |
|----------|-----|
| Репозиторий | GitHub |
| Деплой | Railway из GitHub |
| Запуск | `python main.py` (Procfile / Start Command) |
| Секреты | Variables в Railway |
| Постоянная БД | Volume с Mount Path `/data`, `DATABASE_PATH=/data/augsburg_liquid.db` |

После этого можно коммитить изменения в GitHub — Railway будет пересобирать и перезапускать бота, а база на Volume сохранится.
